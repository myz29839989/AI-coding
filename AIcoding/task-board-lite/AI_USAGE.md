# AI_USAGE.md

## 1. 使用 AI 的目的

本项目使用 AI 辅助完成 Task Board Lite 的需求澄清、系统设计、任务拆解、代码实现建议和测试设计。AI 的作用是提供候选方案和检查清单，最终技术取舍、范围控制和风险判断由我人工确认。

本文件不是 AI 会话原始导出，而是对关键协作过程的整理，重点记录：

- AI 给出的建议。
- 我采纳的部分。
- 我发现并修正的问题。
- 最终形成的工程决策。

## 2. 关键协作过程

### 2.1 需求理解与范围控制

我先让 AI 阅读远程作业需求，并输出项目开发计划、架构设计、技术选型和可行性分析。

AI 初始建议：

- 将系统拆分为前端、后端和数据存储三层。
- 核心实体包括 Team、Task、Status、Member。
- MVP 应优先实现团队创建、任务创建、状态拖拽、看板查询和团队隔离。
- 不建议在 MVP 中加入登录、权限、评论、附件、通知等功能。

人工判断：

- 该建议基本符合题目目标。
- 我确认本项目重点不是功能数量，而是需求拆解、团队隔离、拖拽状态更新和测试覆盖。
- 因此将范围压缩为最小闭环：主控面板 + 团队看板 + JSON 后端 + 自动化测试。

最终决策：

- 主控面板只负责创建团队、进入团队、删除团队。
- 团队页面只负责团队切换、返回主控、创建任务、拖拽更新状态。
- 不做登录、成员权限、任务评论和复杂排序。

### 2.2 数据存储方案调整

AI 初始建议：

- 可使用 SQLite 或 PostgreSQL 保存团队和任务。
- 使用数据库可以更好地保证事务和并发一致性。

实际约束：

- 当前环境中 SQLite 不可用。
- 远程作业更关注工程思路和最小可运行实现，不要求生产级数据库。

人工判断：

- 如果继续依赖 SQLite，会增加本地运行成本。
- 对于小规模 MVP，JSON 文件可以满足演示和测试要求。
- 但必须明确 JSON 不是生产级方案，并说明风险。

最终决策：

- 使用 JSON 文件作为降级存储。
- 数据文件结构为：

```json
{
  "teams": [],
  "tasks": []
}
```

- 使用临时文件替换降低写坏 JSON 的风险。
- 在文档中明确：如果后续进入真实多用户场景，应迁移到 SQLite 或 PostgreSQL。

### 2.3 团队隔离设计

AI 初始建议：

- 使用 `teamId` 区分不同团队的数据。
- 查询看板时按 `teamId` 过滤任务。

我发现的风险：

- 如果状态更新接口只按 `taskId` 查找任务，用户可能用 A 团队页面修改 B 团队任务。
- 团队隔离不能只依赖前端过滤，必须由后端强制保证。

人工修正：

- 所有任务相关接口都必须包含 `teamId`。
- 查询任务、创建任务、更新任务状态都必须使用当前团队上下文。
- 更新任务时必须同时匹配 `teamId` 和 `taskId`。

最终决策：

错误方式：

```js
tasks.find((task) => task.id === taskId)
```

正确方式：

```js
tasks.find((task) => task.id === taskId && task.teamId === teamId)
```

这个规则被写入 `DESIGN.md`，并由自动化测试覆盖。

### 2.4 拖拽状态更新与并发冲突

AI 初始建议：

- 拖拽后调用后端接口更新任务状态。
- 前端刷新看板数据。

我补充的问题：

- 如果两个用户同时拖拽同一个任务，后写入可能覆盖先写入。
- 即使当前是 JSON 文件存储，也要体现基本并发意识。

人工修正：

- 每个任务增加 `version` 字段。
- 前端更新状态时提交当前任务版本。
- 后端校验 `teamId + taskId + version`。
- 版本一致才更新状态并递增版本。
- 版本不一致返回 HTTP 409。

最终决策：

状态更新请求：

```json
{
  "status": "doing",
  "version": 1
}
```

冲突响应：

```json
{
  "error": {
    "code": "VERSION_CONFLICT",
    "message": "task version conflict"
  }
}
```

前端收到 409 后刷新看板，避免静默覆盖其他人的更新。

### 2.5 技术栈降级

AI 初始建议：

- 前端可以使用 React + Vite。
- 后端可以使用 Express。
- 测试可以使用 Jest、Vitest 或 Playwright。

实际问题：

- 当前目标是最短时间实现。
- 不希望依赖安装失败影响项目运行。
- 本地浏览器直接打开页面时，ES Module 方式曾导致页面空白。

人工判断：

- 对于该作业，零依赖可运行比框架完整度更重要。
- 前端可以用原生 HTML、CSS、JavaScript 完成最小看板。
- 后端可以用 Node.js 原生 HTTP 完成 API。
- 测试可以用 Node.js 内置 test runner。

最终决策：

- 前端：静态 HTML + CSS + 原生 JavaScript。
- 后端：Node.js 原生 HTTP。
- 测试：Node.js 内置 `node:test`。
- 不依赖 Express、React、Vite、SQLite。

### 2.6 前端页面设计

AI 初始建议：

- 一个页面中展示团队切换和三列任务看板。

人工调整：

- 页面拆为两个明确区域，降低交互复杂度。

最终设计：

主控面板：

- 创建团队。
- 进入团队。
- 删除团队。

团队看板：

- 团队切换。
- 返回主控。
- 创建任务。
- 三列状态看板。
- 拖拽任务更新状态。
- 显示各列任务数量。

这个设计更符合 MVP，也更便于评审理解。

### 2.7 自动化测试设计

AI 初始建议：

- 至少补充一个后端测试和一个前端测试。
- 后端测试重点覆盖团队隔离或状态更新。

人工判断：

- 当前前端为静态页面，优先用脚本语法检查保证可加载。
- 后端是核心风险点，应重点覆盖团队隔离、状态更新、删除级联和版本冲突。

最终决策：

- 自动化测试脚本放在 `test/` 目录。
- 后端测试覆盖以下场景：
  - 创建团队 A 和团队 B。
  - 团队 A 创建任务。
  - 团队 B 看不到团队 A 任务。
  - 用团队 B 的 `teamId` 更新团队 A 的任务会失败。
  - 正确版本更新任务状态成功。
  - 旧版本更新返回 409。
  - 删除团队后，对应任务不可再访问。

## 3. AI 建议中被修正的点

### 3.1 避免过度依赖框架

AI 曾建议使用 React、Vite、Express 等常见技术栈。考虑到本项目目标是最短时间完成可运行 MVP，我将其降级为零依赖方案。

### 3.2 避免只从前端做团队隔离

AI 的早期描述中强调前端团队切换，但我补充要求后端必须强制 `teamId` 过滤，避免跨团队数据污染。

### 3.3 避免忽略拖拽并发

简单拖拽只更新状态是不够的。我补充了 `version` 乐观锁和 409 冲突处理。

### 3.4 避免提交运行时数据

JSON 文件存储会产生演示数据。我在提交前检查并清空了 `db.json`，避免把本地测试数据作为初始数据提交。

## 4. 最终人工决策汇总

| 决策项 | 最终选择 | 原因 |
| --- | --- | --- |
| 数据存储 | JSON 文件 | SQLite 不可用，MVP 降低环境依赖 |
| 后端 | Node.js 原生 HTTP | 零依赖，可直接运行 |
| 前端 | 静态 HTML/CSS/JS | 无需构建，浏览器可直接打开 |
| 拖拽 | HTML5 Drag and Drop | 满足桌面端 MVP |
| 并发 | 写入队列 + version 乐观锁 | 防止状态更新静默覆盖 |
| 团队隔离 | 后端强制 `teamId` 过滤 | 防止跨团队读写 |
| 测试 | Node.js 内置 test runner | 无需额外测试框架 |
