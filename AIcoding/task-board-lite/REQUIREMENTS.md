# Task Board Lite 需求澄清与假设说明

## 1. 项目目标

Task Board Lite 是一个轻量级任务看板系统，面向多个小团队内部使用。系统目标不是复刻 Trello / Jira 的完整能力，而是在最小范围内实现团队任务管理、任务状态流转、团队数据隔离和基础自动化测试。

本次实现优先满足远程作业的核心考察点：

- 能否正确澄清和拆解需求。
- 能否识别并修正过度复杂或不合理设计。
- 能否完成前后端最小闭环。
- 能否说明 AI 参与设计后的人工判断。
- 能否通过测试覆盖团队隔离、状态更新和并发冲突。

## 2. 用户与使用场景

### 2.1 主要用户

- 小团队成员。
- 项目负责人。
- 面试评审人员。

### 2.2 核心场景

1. 用户进入主控面板。
2. 用户创建一个团队。
3. 用户进入某个团队的任务看板。
4. 用户创建任务，任务默认进入待办列。
5. 用户通过拖拽将任务移动到进行中或已完成列。
6. 用户切换到其他团队，确认任务数据互不混淆。
7. 用户删除团队，系统同步删除该团队下的任务。

## 3. 功能范围

### 3.1 主控面板

主控面板只承担团队入口职责，包含：

- 创建团队。
- 展示团队列表。
- 进入团队。
- 删除团队。

### 3.2 团队任务看板

团队任务看板包含：

- 当前团队展示。
- 团队切换。
- 返回主控。
- 创建任务。
- 三列状态看板：
  - 待办：`todo`
  - 进行中：`doing`
  - 已完成：`done`
- 拖拽任务更新状态。
- 各列任务数量统计。

### 3.3 后端能力

后端提供：

- 团队创建、查询、删除。
- 任务创建。
- 看板查询。
- 任务状态更新。
- 团队数据隔离。
- JSON 文件持久化。
- 简化并发控制。

## 4. 明确不做的范围

MVP 阶段不实现：

- 登录注册。
- 用户权限系统。
- 成员邀请。
- 任务负责人。
- 评论、附件、标签。
- 列内排序。
- WebSocket 实时协同。
- 多实例部署。
- 数据库迁移工具。

这些功能可以作为后续扩展，但不进入当前最短时间交付范围。

## 5. 业务假设

1. 系统不需要登录，但需要通过团队入口进入具体看板。
2. 团队是数据隔离的基本单位。
3. 所有任务都必须属于一个团队。
4. 任务状态固定为 `todo`、`doing`、`done`。
5. 创建任务时默认状态为 `todo`。
6. 拖拽只改变任务状态，不维护列内排序。
7. 删除团队时，同时删除该团队下所有任务。
8. 当前系统面向本地演示和小规模数据。
9. JSON 存储是因 SQLite 当前不可用而采取的降级方案，不作为生产级方案。

## 6. 数据存储降级说明

原计划可使用 SQLite，但当前环境中 SQLite 不可用。为了降低本地运行成本，MVP 改为 JSON 文件存储：

```text
backend/data/db.json
```

JSON 存储可行的原因：

- 当前数据规模小。
- 当前目标是本地演示和作业评审。
- 不需要复杂查询、索引或事务。
- 可以减少环境依赖，提升可运行性。

JSON 存储的限制：

- 不适合高并发。
- 不适合多进程或多实例部署。
- 文件写入过程中存在损坏风险。
- 数据量变大后查询性能会下降。

当前实现通过单进程写入队列、临时文件替换和 `version` 乐观锁降低风险。

## 7. 并发与一致性需求

任务状态更新可能出现并发问题。例如两个用户同时拖拽同一任务：

- 用户 A 将任务从待办移动到进行中。
- 用户 B 同时将任务从待办移动到已完成。

MVP 采用简化方案：

1. 每个任务有 `version` 字段。
2. 前端更新状态时提交当前任务 `version`。
3. 后端校验 `teamId + taskId + version`。
4. 如果版本一致，则更新状态并递增 `version`。
5. 如果版本不一致，则返回 HTTP 409。
6. 前端收到 409 后刷新看板，避免静默覆盖他人修改。

## 8. 验收标准

### 8.1 功能验收

- 可以创建团队。
- 可以进入团队看板。
- 可以删除团队。
- 可以创建任务。
- 可以拖拽任务更新状态。
- 各列数量与任务数量一致。
- 不同团队之间任务数据隔离。
- 删除团队后，该团队任务不可再访问。

### 8.2 技术验收

- 后端可以通过 `npm.cmd start` 启动。
- 前端可以通过浏览器打开 `frontend/index.html` 使用。
- 自动化测试脚本保存在 `test/` 目录。
- 后端自动化测试覆盖团队隔离、状态更新、删除级联和旧版本冲突。

## 9. AI 参与与人工判断

AI 提供了需求拆解、接口设计、并发处理和测试用例建议。

人工干预包括：

- 将数据库从 SQLite 降级为 JSON 文件存储。
- 将前端从 React/Vite 降级为静态 HTML/JS，降低依赖。
- 将后端从 Express 降级为 Node 原生 HTTP，降低安装成本。
- 明确团队隔离必须在后端使用 `teamId` 强制过滤。
- 明确状态更新必须校验 `version`，避免拖拽并发覆盖。
